# 客服FAQ自动分类 — LLM Prompt 优化项目

## 1. 整体优化思路概述

线上客服FAQ自动分类脚本上线后暴露出两大问题：**分类准确率低**、**偶发解析报错**。经排查，根因分布在 Prompt 设计、代码健壮性和安全三个层面。

优化采用"诊断—修复—加固—验证"四步闭环：

```
Code Review 逐行审查
    ↓
Prompt 重设计（类别定义 + 边界规则 + JSON 强约束 + Few-shot）
    ↓
代码加固（解析兜底 + 重试 + 异常隔离 + 并发）
    ↓
30 条边界测试集回归验证 → 100% 准确率
```

**核心改进项：**

| 优化方向 | 原状态 | 改进后 |
|---------|--------|--------|
| Prompt | 无 System Prompt，类别无定义，输出格式约束弱 | 完整 System Prompt，6 类定义 + 边界互斥规则 + JSON 强制输出 + 6 条 Few-shot |
| 输出解析 | 裸文本 `.strip()` 直接使用 | JSON 解析 → 正则兜底 → 白名单校验 → 模糊匹配 → 归入"其他" |
| 异常处理 | API 调用零 try/except | 指数退避重试 3 次 + 单条异常隔离不中断整批 |
| 批处理 | 内存全量收集，最后一次性写盘 | 每 100 条增量写盘 + 线程池并发 |
| 安全 | API Key 硬编码在源码中 | 环境变量注入，缺失直接抛错 |

## 2. Code Review 发现的所有代码问题汇总

### P0 — 会导致程序崩溃 / 频繁报错 / 线上不可用

| # | 严重等级 | 问题 | 代码位置 | 根因分析 | 业务影响 | 修复方案 |
|---|:---:|------|---------|---------|---------|---------|
| 1 | **P0** | **API Key 硬编码在源码中** | `classifier.py:13` | `os.environ.get("DEEPSEEK_API_KEY", "sk-99881...")` 将真实密钥作为默认值写入代码，一旦提交 Git 即永久泄露 | 密钥被盗刷 → API 额度耗尽 → 全部分类请求返回 401，服务不可用 | 移除默认值，改为 `os.environ["KEY"]`，缺失即抛 KeyError |
| 2 | **P0** | **API 调用零异常处理** | `classifier.py:29-37` | `client.chat.completions.create()` 无 try/except，网络超时/429限流/5xx 直接向上抛出 | 网络抖动或服务端波动导致整批任务崩溃，高峰期货运无法及时分类 | 新增 `_call_api_with_retry()`，可重试错误退避 3 次，不可重试错误立即停止 |
| 3 | **P0** | **批处理结果一次性写盘** | `classifier.py:46-57` | 所有结果收集到内存 list，循环全部结束后才 `json.dump` | 中途任意一条崩溃 → 已处理的结果全丢 + 已消耗的 API 费用白费。假设 2000 条第 1998 条报错，前 1997 条全部浪费 | 每 100 条调用 `_save_results` 增量写盘 |

### P1 — 大幅拉低分类准确率 / 业务核心效果失效

| # | 严重等级 | 问题 | 代码位置 | 根因分析 | 业务影响 | 修复方案 |
|---|:---:|------|---------|---------|---------|---------|
| 4 | **P1** | **无 System Prompt** | `classifier.py:28-35` | 分类指令 + 类别列表 + 用户问题全部塞入单条 `user` 消息，LLM 对 user 消息指令遵从度远低于 system 消息 | 输出格式不稳定，模型容易"自由发挥"，是准确率低和格式异常的诱因之一 | 将分类规则、输出格式、边界规则全部写入独立 System Prompt |
| 5 | **P1** | **类别无定义、无边界规则** | `classifier.py:23` | Prompt 仅罗列 6 个类别名称，无任何定义说明。"退款进度查询"该归入退款退货还是物流查询？模型无判断依据 | 边界场景（约占全部问题 15-20%）几乎随机分类，这是准确率低的**核心根因** | 将 `categories.md` 中的完整定义、典型场景、互斥边界全部写入 System Prompt |
| 6 | **P1** | **输出格式约束太弱** | `classifier.py:27` | "只回复类别名称"约束力极弱，模型可能返回 `"退款退货。"`、`"分类结果：退款退货"` 等各种变体 | `.strip()` 无法处理标点和前缀文本，下游精确匹配失败。这是**偶发解析报错的直接根因** | 强制 JSON `{"category":"...","reasoning":"..."}` + 多层解析兜底 |
| 7 | **P1** | **LLM 返回结果无校验** | `classifier.py:37-38` | 裸字符串直接作为分类结果，无任何合法性检查 | 模型可能输出不存在的类别（如"退换货"），下游路由失败 | 白名单校验 → 模糊匹配 → 归入"其他" |
| 8 | **P1** | **输入无校验** | `classifier.py:48` | `item['question']` 直接取用，字段缺失抛 KeyError；空字符串仍调用 API | 脏数据导致批处理中断；空输入浪费 API 调用并产生随机分类 | `item.get("question", "")` 安全取值 + 空检查跳过 API |

### P2 — 隐性隐患 / 可维护性差

| # | 严重等级 | 问题 | 代码位置 | 根因分析 | 业务影响 | 修复方案 |
|---|:---:|------|---------|---------|---------|---------|
| 9 | **P2** | **`print()` 替代 logging** | `classifier.py:59` | 使用 `print()` 而非 `logging` 模块，无日志级别、时间戳、持久化能力 | 线上出问题无法按时间回溯，MTTR 拉长 | 全面改用 `logging` 模块 |
| 10 | **P2** | **串行逐条处理** | `classifier.py:47-54` | 单线程顺序调用 API，30 条耗时约 27s，随数据量线性增长 | 大批量任务耗时长，运营等待时间久 | `ThreadPoolExecutor` 并发处理，支持 `--workers N` 参数 |
| 11 | **P2** | **用户输入直接拼入 Prompt** | `classifier.py:21-26` | f-string 直接嵌入未转义的用户输入 | 恶意用户可构造提示注入操控分类结果 | 当前风险可控（仅用于内部分类），建议后续加入输入清洗 |
| 12 | **P2** | **无进度输出** | `classifier.py:47-54` | for 循环无任何进度日志，100 条和 10000 条表现一样 | 运营无法判断是正常运行还是卡死，盲目杀进程重试浪费时间 | 每 100 条 + 最终完成时输出进度日志 |

## 4. AI 工具辅助开发全过程说明

本项目由 **Claude Code** 辅助完成，完整开发流程如下：

### 4.1 Code Review — 缺陷诊断

- **触发方式**：向 AI 提交 `task1_classifier.py` 源码，要求"找出问题，按严重程度排序"
- **AI 输入**：`task1_classifier.py` + `task1_categories.md` + `task1_prompt.md`
- **AI 输出**：按 P0/P1/P2 三级输出 **12 项缺陷**，每项包含：代码位置、根因分析、业务影响、复现场景
- **人工确认**：审核缺陷分级合理性，确认修复优先级

### 4.2 Prompt 重设计

- **触发方式**：向 AI 提交重设计需求（强化边界、结构化输出、兜底、校验、思考链）
- **AI 输入**：`categories.md`（6 类定义 + 边界规则）+ `task1_prompt.md`（原始 Prompt）
- **AI 输出**：
  - 原始 Prompt 8 项缺陷分析
  - 新版 System Prompt（6 类完整定义、边界互斥规则、JSON 强制格式、6 条 Few-shot 示例）
  - User Message 模板
  - 代码接入方案 + 异常兼容方案
- **人工确认**：审核 Prompt 分类标准是否与业务对齐，Few-shot 示例是否覆盖典型边界场景

### 4.3 代码集成

- **触发方式**：向 AI 提交"把 Prompt 接入代码"
- **AI 输入**：`task1_classifier.py` + 新 Prompt
- **AI 改动**：
  - 新增 `SYSTEM_PROMPT` 常量 + `ALLOWED_CATEGORIES` 白名单
  - 新增 `_parse_response()` 三层解析兜底（JSON 解析 → 正则提取 → 模糊匹配 → 归入"其他"）
  - 消息结构从单 user 消息改为 system + user 双消息
  - 批处理增加 try/except 单条隔离 + `item.get()` 安全取值 + 空输入短路
  - `print()` 全面改为 `logging`
  - `os.environ.get("KEY", "sk-xxx")` 改为 `os.environ["KEY"]`
- **人工确认**：代码审查确认逻辑正确，无引入新问题

### 4.4 追加优化

- **触发方式**：向 AI 提交"基于发现的问题，至少再完成 2 项改进"
- **AI 改动**：
  1. 新增 `_call_api_with_retry()` — 指数退避重试，可重试错误（网络/429/5xx）退避 3 次，不可重试错误（401/403）立即停止
  2. 新增 `ThreadPoolExecutor` 并发批处理 — 支持 `--workers N` 参数，线程安全增量写盘，结果按 `_index` 排序保证输出顺序一致
- **人工确认**：运行测试验证并发正确性和输出一致性

### 4.5 回归验证

- **触发方式**：向 AI 提交"帮我运行脚本"
- **AI 执行**：
  - 安装 openai 依赖，解决 Python 多版本环境问题
  - 运行旧版 1 次 + 新版 1 次，产出 `output_old.json` 和 `output.json`
  - 按 id 对齐逐条对比，产出准确率报告
- **关键发现**：旧版 29/30（#23 边界错分），新版 30/30
- **耗时对比**：旧版串行 ~27s，新版 5 线程并发 ~10s（2.7x）

## 5. 环境配置与代码运行

### 5.1 环境要求

- Python 3.9+
- openai >= 1.0

### 5.2 安装依赖

```bash
pip install openai
```

### 5.3 配置 API Key

**Linux / macOS：**
```bash
export DEEPSEEK_API_KEY="sk-你的密钥"
```

**Windows PowerShell：**
```powershell
$env:DEEPSEEK_API_KEY="sk-你的密钥"
```

**Windows CMD：**
```cmd
set DEEPSEEK_API_KEY=sk-你的密钥
```

### 5.4 批量评估（test_samples.json）

```bash
# 新版 — 默认 5 线程并发
python task1_classifier.py task1_test_samples.json output.json

# 新版 — 自定义线程数
python task1_classifier.py task1_test_samples.json output.json --workers 10

# 旧版 — 对比基线
python task1_classifier_old.py task1_test_samples.json output_old.json
```

### 5.5 单次调用（代码内）

```python
from task1_classifier import classify_question

result = classify_question("我的快递到哪了")
print(result)
# {'category': '物流查询', 'reasoning': '核心诉求是查询包裹位置，匹配物流查询'}
```

### 5.6 输入文件格式

```json
[
  {"id": 1, "question": "我的快递到哪了"},
  {"id": 2, "question": "怎么退货"}
]
```

### 5.7 输出文件格式

```json
[
  {
    "id": 1,
    "question": "我的快递到哪了",
    "predicted_category": "物流查询",
    "reasoning": "核心诉求是查询包裹位置，匹配物流查询"
  }
]
```
